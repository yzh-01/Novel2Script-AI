# 设计理念

本文档回答 YAML Schema 和系统架构中的"为什么这样设计"。

---

## 一、为什么三层分离？（meta / characters / scenes）

剧本创作的标准流程是：故事大纲 → 人物小传 → 场次表。三个阶段的交付物天然独立。

Schema 的三层结构直接对应这个流程：

| Schema 层 | 对应创作阶段 | 职责 |
|-----------|-------------|------|
| `meta` | 故事大纲 | 全局信息：格式、类型、来源 |
| `characters` | 人物小传 | 角色全貌：谁、什么样、和谁关系 |
| `scenes` | 场次表 | 逐一展开：在哪、谁在场、发生了什么 |

**为什么不把角色信息内联到每个场景里？**

小说改编的过程中，一个角色可能出现在 30 个场景中。如果把角色的姓名、性格写进每个场景，改一次角色设定就要改 30 个场景。三层分离 + ID 引用 = 改一处生效全局。这是数据库范式化（第三范式）在剧本领域的应用。

---

## 二、为什么用 discriminatedUnion 处理 blocks？

剧本内容块的三种类型携带完全不同的字段：

| type | 需要 character_id? | 需要 emotion? | 需要 parenthetical? |
|------|:-:|:-:|:-:|
| action | ✗ | ✗ | ✗ |
| dialogue | ✓ | ✓ | ✓ |
| transition | ✗ | ✗ | ✗ |

如果用统一结构（所有字段都设为 optional），三个问题：
1. **语义模糊**：dialogue 忘写 character_id 不报错，写入无效剧本
2. **LLM 混乱**：看到一个 block 有很多 optional 字段，不知道该填哪些
3. **校验无力**：类型安全由运行时人工保证，而非编译期自动保证

`z.discriminatedUnion('type', [...])` 让 Zod 根据 `type` 值自动选择对应的子 Schema，dialogue 缺 character_id 直接报错，action 带 character_id 也报错。从源头阻止了类型混用。

## 三、为什么删掉 character_cue block？

标准剧本格式中，角色入场是写在 action 段落里的。对白前的角色名（dialogue.character_id）本身就是 cue——它告诉演员"轮到你了"。

单独拆出一个 `character_cue` 类型，导致 LLM 在生成时需要判断："这个角色的出场该用 action 还是 character_cue？"——增加了决策负担，输出一致性下降。

4 种 type → 3 种 type，对 LLM 友好，对编辑者也友好。

---

## 四、为什么 time_of_day 只有 day 和 night？

行业标准剧本的 Slug Line 格式是 `INT. 地点 - DAY` 或 `EXT. 地点 - NIGHT`。好莱坞剧本软件（Final Draft、Fade In）也是这样约束的。

具体时间信息（"清晨""黄昏""午后"）放在 `location` 中（如 "北京胡同 - 清晨"），不单独作为枚举值。这样做的好处：
1. 符合行业惯例，生成的剧本可用专业软件（Final Draft 等）导入
2. 不给 LLM 过多选项（7 个值 → 2 个值），减少选择摇摆
3. 减少 Zod 校验复杂度

## 五、为什么保留 source_chapter？

`source_chapter` 是小说转剧本工具的特色字段。没有它，作者看到一段对白，不知道它来自原文哪一章，只能自己翻原文找。

这个字段有三个使用场景：
1. **对照修改**：作者觉得某句对白不对，循 source_chapter 回到原文重读
2. **溯源展示**：章节→场景映射表让改编流程透明化
3. **质量评估**："LLM 第 2 章的改编比第 1 章好"——没有溯源就没法下这种判断

设计为 `int | int[]`（支持数组）是考虑到"一个场景融合了多个章节内容"的改编实践。

## 六、为什么两阶段返回而不做流式（SSE）？

三天开发窗口下，实现成本是关键约束。

| 方案 | 实现成本 | 体验 |
|------|:---:|------|
| 全量等（一次返回） | 1x | 等 30 秒，焦虑 |
| **两阶段（Phase 1 + Phase 2）** | **1.5x** | **先看角色（3 秒），再等剧本** |
| 流式（SSE token-by-token） | 3x | 逐字输出，最流畅 |

两阶段方案：Phase 1 返回角色列表 + 场景大纲（Claude 快速生成 → 2-5 秒），Phase 2 返回完整 YAML（15-30 秒）。用户先确认角色无误的体验，远好于干等 30 秒。但实现成本只是流式的 1/3。

## 七、为什么 CodeMirror 6 而不是 Monaco？

| 维度 | CodeMirror 6 | Monaco (VS Code 内核) |
|------|-------------|----------------------|
| 打包体积 | ~150KB | ~5MB |
| YAML 支持 | @codemirror/lang-yaml 原生 | 需额外插件 |
| 启动速度 | 瞬间 | 需加载 worker |
| 移动端 | 可用 | 基本不可用 |

目标用户是小说作者，不是程序员。CodeMirror 6 的轻量 + YAML 高亮足够满足"看懂并修改 YAML"的需求，Monaco 的代码补全、重构能力是过剩的。

---

## 八、为什么砍掉暗色模式？

三天交付窗口下，暗色模式的 CSS 变量 + 多主题切换 + 测试 ≈ 3 小时。这三个小时用于多类型小说 Prompt 调优或 Schema 文档打磨，对评分的影响远大于暗色模式。

遵循原则：**完成度 > 功能数量**。砍掉不该做的比加新功能更需要勇气。
